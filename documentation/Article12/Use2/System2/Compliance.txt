**Article 12**

**Automated Event Logging System Architecture and Implementation**  
The Pipeline Safety Guardian integrates a comprehensive, automated logging subsystem designed to maintain a continuous and tamper-resistant record of all AI system events throughout its operational life cycle. This logging architecture is embedded at multiple layers including data ingestion, model inference, decision output, and system health monitoring. Logs are generated in real-time by the CNN-based anomaly detection module, the Random Forest fault classifier, and the overall system orchestration components. Each log entry is timestamped with synchronization to a Network Time Protocol (NTP) server ensuring temporal accuracy within ±5 milliseconds. The logging mechanism captures raw input sensor streams metadata, intermediate feature representations, classification outcomes, alert activations, and intervention triggers. This design choice ensures that all system activity relevant to operational safety and fault detection is automatically recorded without manual intervention, thereby enabling granular forensic traceability.

**Identification and Documentation of Risk-Related Situations**  
To align with the need for detecting high-risk situations as per Article 12(2)(a), the system’s logging module explicitly archives events tied to risk manifestation or substantial system modifications. These include detection of anomalous patterns exceeding predefined safety thresholds, failure to classify sensor inputs due to data quality issues, and transitions where model parameters are updated during routine maintenance or retraining cycles. For instance, when the CNN anomaly detector flags a pressure deviation exceeding 3 standard deviations from baseline, an event record contains the exact sensor values, model confidence scores, and system state variables. Equally, logs record versioning information and cryptographic hashes of updated model parameters post any algorithmic retraining on fresh datasets, ensuring a continuous chain of custody for the AI components. These measures were instituted following an internal hazard analysis guided by IEC 61508 principles adapted for AI systems, which identified risk scenarios requiring persistent audit trails.

**Support for Post-Market Surveillance Activities**  
The logging infrastructure is optimized to facilitate post-market monitoring activities as required under Article 12(2)(b). Logged data is securely transferred via encrypted channels (TLS 1.3) at regular intervals to Meridian Safety Systems’ centralized monitoring platform, which supports trend analysis, incident correlation, and long-term system performance evaluation. Logs are stored in an industry-standard append-only format with immutable cryptographic seals to prevent alteration, enhancing data integrity for compliance verifications. Additionally, redaction mechanisms ensure that personal data is excluded to comply with data minimization principles. The post-market surveillance framework leverages logged outputs to detect latent failure modes or emerging operational risks, thereby enabling the provider to issue timely field safety notices or software patches. The system’s capacity to archive and retrieve up to three years of continuous operational logs complies with typical regulatory retention requirements, ensuring availability of historical data for thorough retrospective analysis.

**Operational Monitoring and Real-Time System Oversight**  
In accordance with Article 12(2)(c), the system’s logging supports real-time operational monitoring to uphold continuous safety and reliability. An integrated monitoring dashboard ingests live logs to visualize key performance indicators including sensor data consistency metrics, fault detection rates, false positive and false negative counts, and system uptime. The dashboard incorporates alerting rules configured to trigger escalations whenever suspicious patterns are detected, such as consecutive unclassified events or atypical classifier confidence drops below 70%. These real-time monitoring capabilities enable immediate human-in-loop intervention by pipeline operators and field technicians, consistent with the intended usage scenarios. Logs also document operator acknowledgments and system overrides to maintain accountability. Periodic automated sanity checks record system self-diagnostics outcomes and log anomalies in model processing pipelines, safeguarding against silent failures. This multifaceted logging strategy thus delivers a robust evidential basis for ongoing operational assurance and regulatory oversight.
